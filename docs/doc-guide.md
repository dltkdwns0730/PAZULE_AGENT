# Documentation Guide

> **분류**: 가이드 · **버전**: v2 · **최종 수정**: 2026-04-14
>
> `docs/` 내 모든 문서의 작성 형식과 관리 규칙.

---

## 파일 명명 규칙

- **형식**: `kebab-case.md` (소문자, 하이픈 구분)
- **예시**: `api-specification.md`, `pipeline-visualization.md`
- **금지**: `UPPER_CASE.md`, `snake_case.md`, `camelCase.md`

---

## 필수 헤더

모든 문서의 첫 두 블록은 아래 형식을 따른다.

```markdown
# 문서 제목

> **분류**: <분류> · **버전**: <버전> · **최종 수정**: <YYYY-MM-DD>
>
> 한 줄 설명 — 이 문서가 다루는 내용.
```

### 분류 값

| 분류 | 사용 대상 |
|---|---|
| `설계` | 아키텍처, 기술 결정, 시스템 설계 |
| `레퍼런스` | API 스펙, 파이프라인 다이어그램, 변경 이력 |
| `벤치마크` | 모델 성능 측정, 비교 실험 |
| `전략` | 상용화, 로드맵, 비즈니스 플랜 |
| `가이드` | 작성 규칙, 컨벤션, How-to |

### 버전 형식

- 프로젝트 버전을 따른다: `v2.3`, `v3` 등
- 문서 자체 버전이 별도일 경우 `v1`, `v2`로 독립 관리

## 폴더 구조

```
docs/
├── doc-guide.md              ← 이 파일
├── changelog.md              ← 버전 이력
├── architecture.md           ← 설계 문서 (파이프라인 다이어그램 포함)
├── bugfix-log.md             ← TDD 버그 수정 이력
├── commercialization-plan.md ← 전략 문서
└── benchmarks/               ← 모델 벤치마크
    └── <model>-<scope>.md
```

**규칙:**
- 최상위에는 프로젝트 전반 문서만 위치
- 특정 모델/컴포넌트 관련 문서는 하위 폴더로 분류
- 이미지/데이터 에셋은 `data/assets/`에 위치 (docs/ 내 이미지 금지)

---

## 문서 업데이트 규칙

1. 내용 변경 시 헤더의 `**최종 수정**` 날짜를 반드시 갱신한다
2. 하위 호환되는 수정: 마이너 버전 업 (`v2.3` → `v2.4`)
3. 구조적 변경: 메이저 버전 업 (`v2` → `v3`)
4. README.md Resources 섹션에 새 문서를 추가한다

## 벤치마크 문서 형식

`benchmarks/` 내 파일은 아래 섹션을 포함한다:

```markdown
# <Model> Benchmark — <버전명>

> **분류**: 벤치마크 · **버전**: <v> · **최종 수정**: <날짜>
>
> 모델: `<model-id>` · 장비: <hardware>
> 이미지: `<경로>` (<설명>)

## 목적
## Test 1: ...
## 결론
### 확인된 사실
### 개선 방향
```
