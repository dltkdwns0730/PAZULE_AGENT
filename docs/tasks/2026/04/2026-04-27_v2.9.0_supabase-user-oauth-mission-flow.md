# Task Specification: Supabase User OAuth Mission Flow v2.9.0

## 1. Context & Objective
- **Context:** Mission, coupon, wallet, and profile flows currently use `guest` or client-supplied `user_id`. This makes user ownership spoofable and prevents reliable per-user history after OAuth login.
- **Objective:** Require Gmail/Supabase OAuth before mission participation, use the verified Supabase JWT `sub` as the server-side `user_id`, and keep enterprise login as a later role-based extension.

## 2. Implementation Details
- [x] Add frontend auth state that stores Supabase OAuth session metadata and exposes `userId`, `email`, and `accessToken`.
- [x] Attach `Authorization: Bearer <access_token>` to mission, coupon, profile, and wallet API calls.
- [x] Replace hard-coded `guest` usage in mission flow, profile stats, and coupon wallet with the authenticated user id.
- [x] Require backend auth for user-owned endpoints and prefer verified JWT `sub` over any client-supplied `user_id`.
- [x] Reject mission submit/coupon issue attempts when the authenticated user does not own the mission session.
- [x] Leave `/admin` enterprise access for a later role-based task using `organization_members`.

## 3. Verification & Quality
- **Tests:** `npm test -- supabaseAuth.test.js api.test.js useMission.test.js`, `npm run build`, `uv run pytest tests/api/test_routes.py tests/security/test_auth.py tests/tools/test_supabase_runtime_verify.py -q`, `uv run ruff check app/api/routes.py tests/api/test_routes.py scripts/tools/supabase_runtime_verify.py tests/tools/test_supabase_runtime_verify.py`
- **Manual Validation:** After `VITE_SUPABASE_PUBLISHABLE_KEY` is set, run real Gmail OAuth, confirm `/auth/callback` stores user id, then complete hint -> mission start -> submit -> coupon -> wallet with the same user id.

## 4. Handover Notes
- **Current State:** Completed. User-owned backend endpoints now require Supabase bearer tokens, and frontend mission/profile/wallet flows use the authenticated Supabase user id.
- **Critical Info:** Do not place service-role or `sb_secret_...` values in `VITE_` variables. Backend must treat request body/query `user_id` as untrusted whenever a bearer token is present.
